= ConnectorIO Addons

This repository contains handful set of software integrations which allow to communicate with industrial equipment.
The code which is published allow to plug various hardware (PLC, controllers, meters) to openHAB compatible solutions.
Code in this repository is maintained by *ConnectorIO sp. z o.o.*.

All together openHAB and co7io-addons allows to:

- Enable interaction between different hardware suppliers.
- Begin monitoring of manufacturing processes.
- Collect data and ingest it towards IT systems.
- Securely connect back to the hardware (without VPN client installation).

The top level project structure is divided into four main parts:

 * `bundles` - implementation of integrations
 * `features` - runtime dependency descriptor
 * `kars` - deployment archive for Apache Karaf / openHAB compatible deployments
 * `itests` - integration tests, primarily to verify that KARs can be installed under openHAB

The `bundles` folder have several major groups:

 * `org.connectorio.addons.binding.*` - common/shared code for binding and specific bindings.
 * `org.connectorio.addons.io.*` - various input/output handling code.
 * `org.connectorio.addons.profile.*` - transformations for commands/states exchanged between bindings and items.
 * `org.connectorio.addons.temporal.*` - experimental support for temporal values / schedules.
 * `org.connectorio.addons.config` - a workaround for some trivial shortcomings in (past) openHAB ConfigMapper implementation.

Supported profiles/transformations (module starts with `org.connectorio.addons.profile.xyz`):

|===
|Profile |Module |Description

| `limit-bottom`          | `boundary` | Restrict lowest accepted value (inclusive) for a plain decimal.
| `limit-bottom-quantity` | `boundary` | Restrict lowest accepted value (inclusive) for a quantity with a specified unit.
| `limit-top`             | `boundary` | Restrict highest accepted value (inclusive) for a plain decimal.
| `limit-top-quantity`    | `boundary` | Restrict highest accepted value (inclusive) for a quantity with a specified unit.
| `limit-range`           | `boundary` | Restrict accepted values to a lowest–highest range for a plain decimal.
| `limit-range` (quantity)| `boundary` | Restrict accepted values to a lowest–highest range for a quantity with a specified unit.
| `collector`             | `counter`  | Accumulate incoming values; configures how the first value is handled when the profile is uninitialized.
| `pulse-counter`         | `counter`  | Increment item state by a configurable step on each incoming pulse or tick signal.
| `sustained-counter`     | `counter`  | Variant of pulse-counter that additionally configures uninitialized-state behavior.
| `limit-counter-top`     | `counter`  | Reject counter updates that exceed the current value by more than a configurable anomaly percentage.
| `limit-counter-bottom`  | `counter`  | Guard counter from decreasing below a known state; configures uninitialized-state behavior.
| `toggle`                | `inverse`  | Profile to flip on -> off, off -> on commands/states.
| `isg-value`             | `isg`      | Filter out undefined values returned from modbus registers by ISG heat pumps.
| `isg-energy`            | `isg`      | Scale energy values returned from energy registers polled from ISG heat pumps.
| `cast-binary`           | `cast`     | Cast an incoming value to a binary (OnOff) representation.
| `periodic`              | `timer`    | Send a configurable periodic refresh command at a fixed time interval (seconds up to days).
| `debounce`              | `timer`    | Suppress rapid state changes; forwards only the first or last value within a configurable time window.
| `math-scale`            | `math`     | Multiply values arriving from the device by a scale factor; divide values sent to the device by the same factor.
| `sma`                   | `sma`      | Filter out undefined values returned by SMA inverters for unsupported registers.
| `quantity`              | `quantity` | Attach a physical unit to a plain decimal received from an integration; strips the unit before sending commands back.

|===

The higher level `org.connectorio.addons.profile` module handles chaining of profiles, so they form a small rule, without major runtime overhead.
With ConnectorIO `ProfileFactory` it is possible to have bidirectional transformation i.e. scale [0.1] -> unit [V] for incoming data.
Due to limitation in configuration handling this module can be used only with links produced by XStream Link manager provided by this repository.

Support modules:

* `bddhab` generated REST Api client for openHAB.
* `documentation` textual descriptions, screenshots and such needed to understand operational aspects of working with this code.
* `xsd` XML Schema resources used for offline validation of XML documents.


If you seek for secure VPN-less cloud solution allowing accessing any of above hardware please have a look on https://github.com/connectorio/connectorio-addons[connectorio-cloud] addon.
It allows combining multiple installations, buildings and machines across diverse geographical locations in https://app.connectorio.cloud[ConnectorIO Cloud].
This is a lightweight dashboard application for collected information.

=== Relation to openHAB
This repository contains sources for third party openHAB addons.

This code is deployed independently of openHAB addons repository due to "incompatible" licensing (ASLv2 vs EPLv2).
It will stay here cause it allows us to maintain multiple OH versions and unify infrastructure between multiple bindings.
The shared code for `bindings` which might not be fortunate, simplifies a lot of common operations and reduces boilerplate code.
Quality of code and coding style is slightly different from in openHAB repositories, is not planned to be aligned.
Additionally, code here does not fight against developers as it does not employ @Nullable annotation processor.
